# 产品需求研究 · 方法与规范模板

> 这份文档是「Vibe 产品需求研究 Agent」生成每一份案例报告所遵循的标准方法与规范。
> 新案例(以及内测、对外交付)都应符合这里定义的流程与原则,保证不同报告之间口径一致。

参考案例(均按本规范生成):
- `dist/portable-espresso-maker.html` — 便携咖啡机
- `dist/ai-sleep-earbuds.html` — AI 睡眠耳塞

---

## 1. 一句话定位

输入一个产品想法 → 自动联网深搜**真实用户原声**(Reddit + Hacker News)→ 输出一份
**客观、可交互、可溯源**的需求洞察报告。工具**不替用户下"真/假需求"的判定**,只把证据
和洞察客观呈现,判断交给用户。

---

## 2. 流程(Pipeline)

| 阶段 | 做什么 |
|---|---|
| **SCOPE** | LLM 把想法定范围:`search_idea`、8–12 个相关 subreddit、查询词、6–10 个玩家(含官网) |
| **COLLECT** | Arctic Shift(主) + pullpush(补) + HN,搜 ~999 条相关真实原声,**按规范 ID 跨源去重** |
| **DISCOVER_BRANDS** | 从真实原声里**挖高频品牌**,把开局没列到的(如 Loop)补进玩家名单(数据驱动) |
| **CURATE** | LLM 严格筛相关原声(≥99 条下限);失败自动回退到关键词排序,绝不崩 |
| **CLUSTER** | 聚类成需求主题(含极性 win/pain/neutral) |
| **DECOMPOSE** | VOC 拆解:场景 / 人群 / 未满足需求 / 替代方案 / JTBD / 付费意愿(每条可溯源到原声) |
| **SYNTHESIZE** | 综合 thesis / 核心发现 / 分段洞察 / 切入方向(建议性,非定论)/ 风险 |
| **ASSEMBLE** | 组装 + 品牌去重 + 诚实归类 + 趋势(与语料同源)+ 校验 |
| **TRANSLATE** | 回填目标语言;Reddit 原声始终保留英文原文 |
| **RENDER** | 套用编辑型模板渲染自包含 HTML |

---

## 3. 核心原则(必须遵守)

1. **研究范围要宽。** 想法里的限定词("AI""智能""便携")只当**一个角度**,不是硬过滤。
   研究的是整个**品类 + 底层需求**,要覆盖用户真正会对比的替代品(例:搜"AI 睡眠耳塞"
   也要纳入 Loop 这类被动耳塞)。
2. **玩家数据驱动。** 竞品名单不只靠开局 LLM 猜,要从真实原声里挖高频品牌补充;按品牌
   去重(`Loop Quiet` / `Loop` → 一条;`Soundcore A10` / `A30` → 一条)。
3. **归类要诚实。** 没明确点名品牌的原声归到 `other`,**绝不堆给第一个玩家**(否则虚高
   出假领导者)。口碑好/差率仅在原声 **≥10 条**时才计算,不足标「信息量过少」。
4. **官网直链。** 玩家带 `website`(LLM 确信的官网域名,不确定就留空、**绝不瞎猜**),
   报告里直接链官网「官网 ↗」;拿不到才回退搜索链接。
5. **数据自洽、单一来源。** 所有衍生数据(计数、声量、趋势)都从**同一个去重池**派生,
   不允许某模块单独另查(避免"趋势用 A、正文用 B"打架)。趋势 = 采集原声按年份分布,
   当前年标「截至 <生成日>」的年内数据。
6. **≥99 条真实原声下限**(池子足够时),保证证据厚度。
7. **每个结论可溯源**:原声可点回原帖;VOC 拆解项带 ▸ 证据链接;报告附「全部真实原声」附录。
8. **客观、不下判定**:不在显眼处贴"真/假需求"标签。

---

## 4. 呈现规范(报告)

- **编辑型设计**:纯白底 + 单一深绿强调色(`#0a6b43`)+ 唯一克制陶土红(仅用于负向情绪);
  扁平描边卡、统一字体(Plus Jakarta Sans + Inter)、严格层级、竖向文档流。
- **Top 1–5 序号**:最突出的场景 / 痛点 / 好评 / 差评,均以 Top N 呈现。
- **导出**:右上角「导出」菜单 = 报告 PDF / 原文 Markdown(纯文字、图表文字化) / 全部原声 Markdown。
- **语言**:报告目标语言可选(默认简体中文,含 en/ja/es/fr/de),原声保留英文。
- **移动端**:手机宽度不拥挤、不溢出、字号统一。

---

## 5. 新增案例怎么做

1. 跑生成(本地 app 或脚本),idea 用中文、必要时把 `meta.slug` 固定成干净英文(如
   `portable-espresso-maker`)。
2. 渲染:`python3 scripts/render_report.py <slug>`。
3. 自检:对照本规范第 3 / 4 节逐条过(范围够宽?品牌含挖掘+官网?归类没虚高?可溯源?)。
4. 把 `dist/<slug>.html` 加进搜索页的「查看案例」入口(`scripts/app.py` 的 `FORM_HTML`)。
